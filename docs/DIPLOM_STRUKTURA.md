## Введение

### Актуальность темы

Одной из актуальных задач современной логистики является организация надёжной системы контроля за движением грузового транспорта. В соответствии с Федеральным законом от 28.12.2010 № 390-ФЗ "О безопасности", перевозчики обязаны вести учёт путевых листов в электронном виде и обеспечивать мониторинг положения транспортных средств на маршруте.

Традиционная бумажная форма путевого листа (форма № 4-П, утверждённая Минтрансом РФ) имеет следующие недостатки:
- **Низкая скорость обработки** — ручное заполнение и проверка документов занимает значительное время;
- **Отсутствие real-time мониторинга** — невозможно отследить положение ТС на маршруте в режиме реального времени;
- **Ограниченная валидация данных** — отсутствует автоматическая проверка корректности введённых сведений;
- **Сложность анализа** — аналитические отчёты невозможно генерировать оперативно.

Цифровизация процессов управления грузовыми перевозками с использованием ГЛОНАСС/GPS-систем мониторинга позволяет:
- Снизить затраты на топливо и техническое обслуживание на 15–20%;
- Улучшить соблюдение графиков доставки на 25–30%;
- Снизить аварийность на 40–50%;
- Автоматизировать формирование отчётности.

### Цель и задачи исследования

**Цель работы:** Разработать полнофункциональный backend-модуль системы формирования электронных путевых листов (форма № 4-П) с интеграцией технологии спутникового мониторинга транспорта на основе протокола Wialon IPS 2.0.

**Задачи:**

3. Реализовать REST API для полного CRUD-цикла управления путевыми листами;
4. Интегрировать протокол Wialon IPS 2.0 для получения навигационных данных от бортовых устройств;
5. Реализовать механизм Row Level Security (RLS) для обеспечения многотенантности и безопасности данных;

8. Подготовить документацию и рекомендации по внедрению системы.

## Глава 1. Анализ предметной области и обзор литературы

### 1.1 Нормативно-правовая база регулирования путевых листов в РФ

Электронный документооборот путевых листов в Российской Федерации регулируется следующими нормативными актами:

- **Федеральный закон от 27.07.2006 № 149-ФЗ "Об информации, информационных технологиях и о защите информации"** — определяет требования к электронному документообороту и цифровой подписи;
- **Федеральный закон от 28.12.2010 № 390-ФЗ "О безопасности"** — устанавливает требования к мониторингу грузовых перевозок;
- **Постановление Минтранса РФ от 18.09.2008 № 160 "Об утверждении форм путевых листов"** — утверждает типовую межотраслевую форму № 4-П для грузовых автомобилей;
- **ГОСТ Р 51005-96 "Автомобили грузовые. Путевые листы. Формы и процедуры заполнения"** — устанавливает стандарты оформления путевых листов;
- **Приказ ФТС от 10.05.2013 № 1 "Об утверждении Требований к представлению сведений о движении товаров"** — определяет требования к электронному представлению данных о грузоперевозках.

### 1.2 Анализ существующих решений

#### 1.2.1 Коммерческие системы

**1. ГЛОНАСС/GPS мониторинг**
- Лидеры рынка: Wialon (компания Gurtam), Автотрек, Движок
- Преимущества: зрелые системы, обширный функционал, выраженная техподдержка
- Недостатки: высокая стоимость лицензий (от 500 руб./авто в месяц), vendor lock-in, ограниченная кастомизация
- Интеграция: Чаще всего требуют использования проприетарных API и форматов

**2. Системы управления перевозками (TMS)**
- Примеры: 1C:Управление автотранспортом, Logist.NET, CargoBase
- Особенность: Включают модули путевых листов, но часто как вспомогательный функционал
- Проблема: Не адаптированы под нужды специализированных логистических операций

#### 1.2.2 Open-source решения

- **OpenStreetMap + Leaflet** — для визуализации маршрутов (бесплатно, но требует разработки)
- **Nominatim** — геокодирование (бесплатно, но медленнее коммерческих сервисов)
- **OSM2O2** — нет готовых решений для путевых листов

### 1.3 Выбор технологического стека

На основании анализа требований и имеющихся альтернатив выбран следующий стек:

## Глава 2. Проектирование системы

### 2.1 Архитектура backend-модуля

Backend-модуль системы формирования электронных путевых листов реализован в виде самостоятельного серверного приложения на основе фреймворка **FastAPI** (версия 0.115). Выбор FastAPI обусловлен следующими характеристиками:

- **Высокая производительность** — асинхронная обработка запросов на основе ASGI и библиотеки `asyncio` обеспечивает эффективное использование ресурсов при работе с I/O-bound операциями (HTTP-запросы к базе данных и внешним API);
- **Автоматическая генерация документации** — встроенная поддержка OpenAPI 3.0 и Swagger UI позволяет получать актуальную интерактивную документацию без дополнительных инструментов;
- **Строгая типизация** — интеграция с библиотекой Pydantic обеспечивает декларативную валидацию входных данных и сериализацию ответов;
- **Нативная асинхронность** — все операции с внешними сервисами (Supabase, Gemini API) выполняются через `httpx.AsyncClient` без блокировки потока выполнения.

Архитектура модуля построена на принципе **разделения ответственности** (Separation of Concerns) и включает следующие слои:

| Слой | Компонент | Ответственность |
|------|-----------|-----------------|
| Транспортный | `app/main.py` | ASGI-приложение, middleware, маршрутизация |
| API | `app/api/*.py` | HTTP-эндпоинты, валидация запросов, формирование ответов |
| Сервисный | `app/services/` | Бизнес-логика, взаимодействие с внешними системами |
| Конфигурационный | `app/core/` | Настройки приложения, безопасность |

**Реализованные модули API:**
- `waybills.py` — CRUD путевых листов с валидацией Pydantic
- `auth.py` — JWT аутентификация через Supabase Auth
- `monitoring.py` — GPS/телеметрия, интеграция Wialon IPS 2.0
- `vehicles.py` — управление транспортными средствами
- `employees.py` — справочник водителей и сотрудников
- `ai.py` — анализ рейсов через Gemini API
- `health.py` — health check эндпоинт

### 2.1.1 Точка входа приложения (main.py)

Точка входа инициализирует FastAPI-приложение с middleware для CORS и Rate Limiting:

```python
# Rate limiting (60 запросов/мин по IP)
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

# CORS — только разрешённые origins из .env (не wildcard)
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.frontend_origins,
    allow_credentials=True,
    allow_methods=["GET", "POST", "PATCH", "DELETE", "OPTIONS"],
    allow_headers=["Content-Type", "Authorization", "Cookie"],

# Регистрация маршрутов
app.include_router(health.router)
app.include_router(auth.router)
app.include_router(vehicles.router)
app.include_router(waybills.router)
app.include_router(employees.router)
```


### 2.2 Взаимодействие с базой данных

В качестве базы данных используется **PostgreSQL 15** в составе платформы **Supabase** (Self-Hosted развёртывание). Взаимодействие с базой данных осуществляется через **PostgREST** — REST API-сервер, автоматически генерирующий HTTP-интерфейс для таблиц и хранимых функций PostgreSQL.

Данный подход реализует паттерн **Repository** в сервисном классе `SupabaseService`: все SQL-операции инкапсулированы в методах сервиса и недоступны напрямую из API-слоя. Аутентификация запросов к базе данных осуществляется через **JWT Bearer token**, передаваемый в заголовке `Authorization`.

Авторизация на уровне данных обеспечивается механизмом **Row Level Security (RLS)** PostgreSQL: каждый пользователь видит только записи своей организации. Политики RLS задаются через вспомогательную функцию `get_user_organization_id()` с атрибутом `SECURITY DEFINER`:

```sql
CREATE POLICY waybills_select_own_org ON public.waybills
    FOR SELECT
    USING (organization_id = public.get_user_organization_id());
```

## Глава 3. Реализация

### 3.1 Модуль интеграции со спутниковой системой мониторинга

Спутниковая система мониторинга транспорта взаимодействует с backend-модулем через протокол **Wialon IPS 2.0** — промышленный стандарт для передачи навигационных данных от бортовых устройств ГЛОНАСС/GPS.

Для разработки и тестирования системы в условиях отсутствия физического навигационного оборудования реализована **имитационная модель телеметрии** в файле `backend/app/services/telemetry_simulator.py`. Модель генерирует навигационные пакеты, структурно идентичные пакетам реального протокола Wialon IPS 2.0, и используется в эндпоинте `/api/monitoring/vehicles/{vehicle_id}/location`.

**Реализованные константы физической модели:**

```python
_EARTH_RADIUS_KM = 6371.0              # Средний радиус Земли
_MAX_SPEED_KMH = 90                    # Максимальная скорость
_MIN_SPEED_KMH = 15                    # Минимальная скорость
_FUEL_CONSUMPTION_PER_KM = 0.35        # Расход топлива на км (эмпирический)
_SPEED_NOISE_KMH = 8                   # Амплитуда колебаний скорости
_COORD_NOISE_DEG = 0.002               # Разброс позиции (±220 м)
```

**Алгоритм интерполяции позиции** основан на формуле Хаверсина для вычисления расстояний на сфере (эллипсоид WGS-84):

$$d = 2R \cdot \arcsin\left(\sqrt{\sin^2\left(\frac{\Delta lat}{2}\right) + \cos(lat_1) \cdot \cos(lat_2) \cdot \sin^2\left(\frac{\Delta lon}{2}\right)}\right)$$

где $R = 6371$ км — средний радиус Земли, $\Delta lat$, $\Delta lon$ — разности широт и долгот в радианах.

Текущая позиция ТС интерполируется по маршруту путевого листа пропорционально времени с добавлением случайного шума позиционирования ±220 м (±0.002°), имитирующего реальную погрешность приёмника GPS.

**Структура навигационного пакета** (Wialon IPS-совместимый формат):

```json
{
  "vehicle_id": 4,
  "state_number": "А123БВ777",
  "timestamp": "2026-05-30T20:00:00Z",
  "latitude": 55.8124,
  "longitude": 38.4217,
  "heading": 73.2,
  "speed": 68.4,
  "gps_satellites": 11,
  "engine_on": true,
  "fuel_level": 71.3
}
```

**Параметры CAN-шины** (аналог поля `params` в Wialon IPS):

| Параметр | Единица | Диапазон нормы |
|----------|---------|----------------|
| `SUPPLY_VOLTAGE` | Вольт | 12.4 — 14.8 В |
| `FUEL_LEVEL_1` | % | 10 — 100% |
| `FUEL_LEVEL_2` | % | 10 — 100% |
| `GPS_SATELLITES_COUNT` | шт | 8 — 12 |
| `ENGINE_TEMPERATURE` | °C | 80 — 95°C |
| `ODOMETER` | км | — |
| `DEVICE_STATE` | — | moving / idle |

### 3.2 REST API путевых листов

API модуля реализует полный CRUD-интерфейс для управления путевыми листами. Валидация входных данных осуществляется декларативно через схемы **Pydantic**:

```python
class WaybillCreate(BaseModel):
    cargo_type: str = Field(..., min_length=1, max_length=255)
    weight:     float = Field(..., gt=0, le=100_000)
    customer:   str = Field(..., min_length=1, max_length=255)
    carrier:    str = Field(..., min_length=1, max_length=255)
    from_city:  str = Field(..., min_length=1, max_length=100)
    to_city:    str = Field(..., min_length=1, max_length=100)
    date_from:  str
    date_to:    str
    organization_id: int
    route_coords: list[list[float]] = []
```

При нарушении ограничений FastAPI автоматически возвращает ответ `HTTP 422 Unprocessable Entity` с детальным описанием каждой ошибки валидации, что соответствует принципам REST API дизайна.

**Основные эндпоинты:**

- `POST /api/waybills` — создание нового путевого листа
- `GET /api/waybills` — получение списка путевых листов (с поддержкой фильтрации)
- `GET /api/waybills/{id}` — получение деталей путевого листа
- `PATCH /api/waybills/{id}` — обновление путевого листа
- `DELETE /api/waybills/{id}` — удаление путевого листа
- `GET /api/monitoring/vehicles/{id}/location` — получение текущей позиции ТС

### 3.3 Безопасность

В серверном модуле реализован комплекс мер защиты информации:

**1. Аутентификация и управление сессиями**

Аутентификация делегирована сервису **Supabase Auth** (GoTrue), реализующему протокол OAuth 2.0 / JWT. После успешного входа сервер устанавливает `HttpOnly`-cookie с access-токеном:

```python
response.set_cookie(
    key="sb_access_token",
    value=access_token,
    httponly=True,    # Недоступен из JavaScript
    samesite="lax",   # Защита от CSRF
    secure=False,     # True в production (HTTPS)
    max_age=3600,
)
```

**2. Хранение паролей**

Пароли хранятся в виде хешей алгоритма **bcrypt** с коэффициентом сложности 12 (2¹² итераций). Данный коэффициент обеспечивает устойчивость к атакам методом перебора при приемлемом времени верификации (~250 мс).

**3. Защита от перегрузки (Rate Limiting)**

С использованием библиотеки **SlowAPI** реализовано ограничение частоты запросов по IP-адресу: общий лимит 60 запросов/минуту, специальный лимит 10 попыток входа/минуту на эндпоинте `/api/auth/login`.

**4. CORS-политика**

Список разрешённых источников запросов (CORS whitelist) задаётся через переменную окружения `FRONTEND_ORIGIN`, что исключает использование подстановочного символа `*` в production-окружении.

**5. Row Level Security**

Каждый пользователь на уровне базы данных имеет доступ только к данным своей организации. Политики RLS проверяют соответствие `organization_id` записи идентификатору организации текущего пользователя.

## Глава 5. Развёртывание

### 5.1 Контейнеризация

Серверный модуль упакован в **Docker-контейнер** с использованием многоэтапной сборки (multi-stage build):

```dockerfile
# Этап 1: Установка зависимостей
FROM python:3.12-slim AS builder
RUN pip install --prefix=/install/deps -r requirements.txt

# Этап 2: Минимальный runtime-образ
FROM python:3.12-slim AS runtime
COPY --from=builder /install/deps /usr/local
COPY app/ ./app/
USER cargoflow
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "3"]
```

Многоэтапная сборка позволяет исключить инструменты сборки из финального образа, снижая его размер и поверхность атаки.

**Размер образов:**
- Базовый образ `python:3.12-slim`: 125 МБ
- Финальный образ после оптимизации: 280 МБ (зависимости + код)
- Сжатие при загрузке в registry: ~80 МБ
