# Supabase + FastAPI: как подключать и как развивать

## 1. Что сейчас уже подключено

Подключение к Supabase реализовано через:

- [supabase_http.ts](../supabase_http.ts) — HTTP-клиент для auth и REST;
- [server.ts](../server.ts) — Express API, который использует этот клиент.

Текущий рабочий сценарий:

- логин через Supabase Auth;
- запросы к /rest/v1/vehicles и другим таблицам;
- cookie-based session в BFF.

## 2. Почему это работает сейчас

Потому что:

- в Supabase есть готовый Auth endpoint;
- REST API доступен через /rest/v1/;
- в коде уже есть базовый клиент с anon key и bearer token.

## 3. Что нужно для production

Перед запуском в проде стоит:

- перевести ключи в .env;
- включить RLS и проверить политики доступа;
- убрать локальный fallback, если он больше не нужен;
- использовать service role только внутри защищённого backend.

## 4. Как выглядит FastAPI-вариант

Вариант с FastAPI выглядит так:

```mermaid
flowchart LR
    A[React Frontend] --> B[Express BFF]
    B --> C[FastAPI Service]
    C --> D[Supabase DB/Auth]
```

Где:

- Express — UI gateway, cookies, auth bridge;
- FastAPI — бизнес-логика, CRUD, отчёты, печать, аналитика;
- Supabase — хранение данных и аутентификация.

## 5. Рекомендуемая граница ответственности

### Express/BFF

- принимать запросы от фронтенда;
- работать с cookie и авторизацией;
- проксировать простые вызовы в Supabase;
- не хранить сложную бизнес-логику.

### FastAPI

- CRUD по грузам, водителям, путевым листам;
- расчёты статусов, маршрутов, аналитики;
- интеграция с телеметрией и AI;
- подготовка данных для UI.

## 6. Практический план миграции

1. Сначала оставить Express как основной BFF.
2. Выделить FastAPI только для сложных модулей.
3. Перенести бизнес-правила из Express в FastAPI.
4. Упростить фронтенд до вызова одного чистого API.

## 7. Рекомендация

Да, FastAPI — хороший вариант для бэкенда, но не обязательно сразу с нуля.
Сейчас лучше:

- оставить текущий Node/Express вариант как рабочий BFF;
- постепенно переносить логику в FastAPI по мере роста проекта.
